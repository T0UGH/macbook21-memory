# 2026-03-15 团队版 Claude Code 业务知识与 P0 命中方案

## 背景

当前团队情况：
- 约 20 人开发团队
- 统一使用 Claude Code 做开发
- 技术栈以 Go 后端微服务为主
- 每个仓库都不大
- 已经具备：服务级 `CLAUDE.md`、跨仓库级 `CLAUDE.md`、以及 rules

当前真正缺少的，不是规则入口，而是：

1. **团队内部 P0/P1 关键知识，如何在关键时刻让 Claude Code 命中**
2. **如何借鉴 claude-mem 的思路，做团队共享记忆/业务知识沉淀**

---

## 一、总体判断

这两个问题都非常有意义，但优先级不同。

### 1. P0 知识命中问题
优先级最高。

它解决的是：
- 事故红线
- 合规/安全要求
- 架构禁区
- 高风险改动约束
- 关键排障路径

这是“让 Claude 在关键时刻不犯大错”的问题。

### 2. 团队共享记忆问题
也重要，但更需要治理。

它解决的是：
- 经验复用
- 跨人协作
- 减少重复探索
- 新人上手加速

但这类系统更容易变成噪音库，因此不应该一开始就全自动共享所有 observation。

---

## 二、对当前阶段的判断

团队其实已经有了：
- 规则入口层（服务级/跨仓库 `CLAUDE.md`）
- 基础规则层（rules）

所以现在**不缺入口**，真正缺的是：

### 业务知识沉淀层
也就是：
- 为什么这么设计
- 哪些业务规则不能碰
- 哪些线上行为和代码表面不一致
- 哪些坑是业务语义导致的，不是技术语义导致的
- 某些事故的根因为什么必须结合业务才能理解

一句话：

**团队当前缺的是“业务语义层”，不是“更多 rules”。**

---

## 三、团队版 Claude Code Memory 的建议分层

建议分 4 层：

### 第 1 层：P0/P1 硬知识层
内容：
- 事故红线
- 架构禁区
- 安全/合规规则
- 数据一致性约束
- 高风险改动规则
- 关键排障路径

特点：
- 少
- 准
- 强约束
- 必须命中

本质上更像 **组织级规则库**。

### 第 2 层：Repo / Service 知识层
内容：
- 服务职责
- 关键模块
- 数据流
- 常见坑
- 本服务历史事故
- 调试入口
- 关键 schema / topic / job 说明

特点：
- 与 repo 强绑定
- 默认应该被 Claude 理解

### 第 3 层：团队经验层
内容：
- 已验证排障路径
- 架构 trade-off
- 性能优化经验
- 某类问题常见根因
- 跨服务联动经验

特点：
- 有复用价值
- 不是强规则
- 必须设置质量门槛

### 第 4 层：个人/会话工作记忆层
内容：
- 当前 session 的 observation
- 阶段 summary
- 最近做了什么
- 下次回来怎么接上

特点：
- 短期
- 高动态
- 更适合个人或 repo 内使用
- 不适合全量团队共享

---

## 四、对团队共享的核心建议

### 不建议一上来做
**“全团队共享所有 observation”**

原因：
- 噪音太大
- 结论未验证
- 跨服务污染
- 旧经验误导新任务
- 成本和治理复杂度会迅速失控

### 正确顺序

#### Phase 1：先做 P0/P1 知识命中
目标：让 Claude 在关键时刻不犯大错

#### Phase 2：做 repo/service 知识层
目标：让 Claude 对当前服务有稳定背景认知

#### Phase 3：再做团队共享经验层
目标：复用团队验证过的经验

#### Phase 4：最后接个人/会话级记忆
目标：增强连续工作流体验

---

## 五、P0 知识命中系统的建议方案

### 目标
不是做一个“大知识库”，而是：

**在高风险时刻，把最该被看到的组织知识，稳定地送到 Claude Code 面前。**

### P0 知识的范围
建议只收这 5 类：

1. 事故红线
2. 架构禁区
3. 安全/合规规则
4. 高风险改动约束
5. 关键排障路径

标准：

**如果 Claude 忘了这条，是否可能造成严重事故、数据错乱、安全问题或重大返工？**

如果不会，不要进 P0。

---

## 六、不要怎么做

不要做成：
- 大而全 wiki
- 一堆没人维护的长文档
- 没标签的 Markdown 堆
- 谁都能随意加、没人 review 的经验池

否则结果一定是：
- 命中差
- 噪音大
- 很快过时

---

## 七、P0 知识的正确落地方式

建议采用三段式：

### 1. 知识对象标准化
每条 P0 知识都做成**规则卡片**，而不是随笔。

建议字段：
- 标题
- 适用范围
- 触发条件
- 规则内容
- 错误后果
- 正确做法
- 例子
- owner
- 更新时间

### 2. 命中机制
建议三种触发：

#### A. repo/service 触发
进入某服务时，默认加载该服务最高优先级的 3~5 条规则

#### B. prompt 语义触发
用户 prompt 命中相关主题时，补充相应规则

例如：
- 数据库 schema
- 重试
- 幂等
- 缓存
- 登录逻辑
- 支付回调

#### C. path/file 触发
Claude 读/改到某些高风险目录或文件时，再触发相关规则

例如：
- `internal/payment/`
- `internal/auth/`
- `dao/`
- `migration/`
- `job/settlement/`

### 3. 注入策略
不是全量塞入，而是：

- 默认每个服务注入 3~5 条最高优先级规则
- prompt 命中时再补 1~3 条
- path 命中时再补 1~2 条

原则：
- 宁少勿多
- 一次不要超过约 10 条
- 每条都必须是 Claude 可直接执行的 briefing

## 八、如何融入 Claude Code 生态

### 一个知识一个 skill？
**不建议。**

原因：
- skill 粒度太重
- 维护会爆炸
- 检索/组合能力差
- 单条知识更适合做卡片，而不是做 skill

### 正确做法

#### skill 的角色
skill 应该做：
- 知识路由器
- 规则引擎
- 检索器
- briefing 生成器

而不是知识本体。

#### 知识本体放哪里
放在：
- 团队知识库 repo
- Markdown / YAML frontmatter 卡片
- 按服务 / 模式 / incident 分类

#### Claude Code 接入方式
- SessionStart：默认加载该服务最核心规则
- UserPromptSubmit：按 prompt 关键词命中业务/风险卡片
- PostToolUse / 路径命中：在真正改高风险代码时再次补充相关规则

一句话：

**自动命中靠 hooks，知识本体靠知识库，skill 只做路由。**

---

## 九、关于 hooks 的作用判断

结论：

**如果要做到“关键时刻自动命中”，就仍然会很依赖 hooks。**

因为 hooks 才能感知：
- 进入 repo
- 用户发 prompt
- Claude 正在读/改哪个路径
- 工具访问了哪些危险区域

但也不是“只靠 hooks”。

更准确说：
- hooks 负责触发时机
- 知识库负责内容存储
- skill/检索器负责查找和组装

---

## 十、关于 `CLAUDE.md` 的判断

### 问题：直接把这些知识放到服务的 `CLAUDE.md` 里行不行？

结论：

**可以，而且这是一个非常好的 0→1 起步方案。**

### 为什么可行
- 离代码近
- Claude Code 天然会读
- 实现成本最低
- 服务边界清晰
- 责任人明确

### 但它有边界
`CLAUDE.md` 只适合放：
- 本服务最核心的 P0/P1 知识
- 最稳定、最高频的规则
- 最重要的业务入口说明

不适合放：
- 大量 incident 明细
- 大量跨服务经验
- 大量变化快的业务知识
- 团队完整知识系统

### 正确演进

#### Phase 1
把每个服务最关键的 P0/P1 放进 `CLAUDE.md`

#### Phase 2
当内容变多时，拆出：
- `docs/domain/`
- `docs/business-rules/`
- `docs/incidents/`
- `docs/decisions/`

而 `CLAUDE.md` 只保留入口和最关键的 3~5 条内容。

#### Phase 3
再引入：
- 统一团队知识库
- hooks 动态命中
- skill 检索/路由
- 团队共享记忆

一句话：

**`CLAUDE.md` 适合做 P0 知识的第一落点，但不适合单独承载长期团队知识系统。**

## 十一、当前团队最缺什么

结论非常明确：

你们已经有了：
- 规则层
- 入口层

但缺的是：
- 业务语义层
- 事故经验层
- 决策背景层

也就是：
- 为什么这么设计
- 哪些业务规则不能碰
- 某些代码表面含义和线上真实含义为什么不一致
- 某些状态机和字段为什么不能按字面理解
- 某些事故的根因为什么必须结合业务理解

一句话：

**团队当前真正缺的不是“规矩”，而是“业务知识沉淀层”。**

---

## 十二、建议沉淀的业务知识类型

建议按 4 类组织业务知识卡片：

### 1. 领域概念卡
解释业务概念。

例如：
- 什么是预占单
- 什么是补单
- 什么是冻结态
- 什么叫部分成功

### 2. 业务规则卡
解释业务规则。

例如：
- 什么情况下允许退款
- 什么情况下禁止重试
- 为什么某状态不可逆
- 某字段更新前置条件是什么

### 3. 事故/坑卡
解释真实踩坑。

例如：
- 重复扣款的真实原因
- 某类库存错乱不是事务问题而是业务幂等缺失
- 某类“脏数据”其实是对账窗口延迟导致

### 4. 决策/约束卡
解释为什么这么设计。

例如：
- 为什么不用强一致
- 为什么这里允许最终一致
- 为什么拆成两个 topic
- 为什么读写分离不能用于这个查询

这些内容不建议继续堆进 `CLAUDE.md`，更适合放在：
- `docs/domain/`
- `docs/business-rules/`
- `docs/incidents/`
- `docs/decisions/`

---

## 十三、如何让 Claude Code 在合适时机读到这些业务卡片

关键不是“存在哪里”，而是 **触发机制**。

建议使用三层命中：

### 1. 默认命中：进仓库就读
适合：
- 服务概览
- 核心业务对象
- 最重要的状态机
- 3~5 条最高优先级业务规则

做法：
- 服务级 `CLAUDE.md` 提供入口索引
- 指向 `docs/domain/`、`docs/business-rules/`
- Claude 进入 repo 时先读这份入口

### 2. Prompt 命中：用户问到某主题时读
适合：
- 退款
- 补单
- 对账
- 重试
- 状态流转
- 风控
- 结算

做法：
- 给每张卡片加 tags / keywords
- 用户 prompt 命中时，检索并注入对应业务卡片

### 3. Path 命中：改到某目录/文件时读
适合：
- 特定 handler
- 状态机代码
- 核心 job
- payment/auth/order 等高风险目录

做法：
- 每张卡片绑定 paths/files
- Claude 读/改这些路径时自动补卡片

### 三层组合原则
- 基础背景默认给
- 任务相关卡片按 prompt 命中
- 高风险区域按 path 命中

一句话：

**不是让 Claude 每次都读所有卡片，而是在三个关键时机读到最相关的几张卡片。**

---

## 十四、当前阶段的最优路线

对于当前团队，不建议一步到位做复杂的团队共享 memory 系统。

建议路线：

### Step 1
继续保留现有服务级/跨仓库 `CLAUDE.md` 与 rules 作为入口层

### Step 2
新增一层业务知识卡片：
- `docs/domain/`
- `docs/business-rules/`
- `docs/incidents/`
- `docs/decisions/`

### Step 3
服务级 `CLAUDE.md` 只保留：
- 最关键 3~5 条 P0/P1
- 业务知识索引入口

### Step 4
后续再通过 hooks + skill + 路由器做动态命中

### Step 5
当知识卡片质量稳定后，再考虑团队共享记忆层

---

## 十五、最终总判断

这次讨论形成的几个明确结论：

1. 团队当前最缺的是**业务知识沉淀层**，不是更多 `CLAUDE.md` 或 rules。
2. P0 知识命中系统非常值得先做，且优先级高于团队共享记忆。
3. 不应该“一条知识一个 skill”，而应该由少数 skill/检索器去路由知识库。
4. 要做到自动、关键时刻命中，就仍然需要 hooks。
5. `CLAUDE.md` 非常适合作为 P0 知识的第一落点，但不能承载完整团队知识系统。
6. 业务知识应该沉淀成结构化卡片，而不是继续写散乱文档。
7. Claude Code 最合适的读取方式是：
   - 默认命中
   - prompt 命中
   - path 命中
8. 最合理的演进顺序是：
   - 规则入口层（已有）
   - 业务知识卡片层（最缺）
   - 动态命中层
   - 团队共享经验层
   - 更复杂的共享 memory

---

## 一句话总结

**你们现在不该继续堆更多规则入口，而应该补一层“业务语义/事故经验/决策背景”的结构化知识卡片，再通过 `CLAUDE.md` + hooks + 少量 skill，在合适时机把最相关的卡片喂给 Claude Code。**
